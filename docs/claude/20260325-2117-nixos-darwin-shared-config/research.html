Entry Point
flake.nix
Minimal entry point. Declares inputs and delegates everything to import-tree ./modules. No manual import lists.
Flake Inputs
  • nixpkgs nixos-unstable
  • flake-parts module framework
  • import-tree auto-discovery
  • wrapper-modules portable wrapping
  • home-manager user config
  • nix-darwin macOS system
every .nix file is a flake-parts module
Dendritic Module Layer

Every .nix file under modules/ is a uniform flake-parts module. Files organize by feature, not by configuration class. A single file can define NixOS config, darwin config, home-manager config, and packages.

modules/features/
Feature-oriented modules. Each file owns one concern across all platforms. Wrapped packages are testable standalone via nix run.
niri.nix noctalia.nix git.nix shell.nix terminal.nix editor.nix ssh.nix
modules/hosts/
Per-machine configuration. Composes feature modules into host-specific systems.
linux-desktop/ linux-laptop/ macos-laptop/
modules/os/
Cross-host shared settings per platform class.
nixos.nix darwin.nix nix-settings.nix
modules produce shared & platform-specific outputs

Shared Across Platforms

wrapper-modules

Produces portable wrapped derivations via perSystem. No dependency on home-manager or NixOS. Same package on both platforms.

  • myNiri compositor with keybindings linux only
  • myNoctalia shell with settings JSON linux only
  • myTerminal terminal emulator config
  • myEditor editor with plugins & settings
home-manager

User-level configuration via flake.modules.homeManager. Loaded as NixOS module or darwin module. Same underlying modules.

  • git identity, aliases, signing
  • shell zsh/fish, starship, direnv
  • ssh keys, hosts, agent
  • xdg directories, MIME associations

Secrets & Public Repo Strategy

Config Files: Inline, Not Submodules

Config files from ~/.dotfiles are copied into the NixOS repo directly. Private Sourcehut repos remain as historical source — not runtime dependencies. Repo is fully self-contained and public.

  • Each feature module co-locates .nix wiring with native config files
  • Audit private repos for secrets before making public
  • Optional: _local/ gitignored dir for truly private overrides
sops-nix: Encrypted Secrets

Secrets encrypted with age keys, decrypted at activation time to /run/secrets/ (tmpfs). Env var secrets sourced by bash from the decrypted file.

  • secrets/common.yaml shared across all hosts
  • secrets/home.yaml user-level (git tokens, API keys)
  • sops.templates compose env var files from individual secrets
  • Bootstrap: place age key, then single-command deploy
Security Audit Results

No hardcoded secrets found — all passwords/tokens use op (1Password CLI). However, several files contain personal identifiers and work-specific config.

sops-encrypt
  • aerc/accounts.conf email identity
  • senpai/senpai.scfg IRC identity

  • scripts/linux/romm server config

Testing Strategy (No System Overwrite)

Safe Iteration Workflow
1. Validate & Build
  • nix flake check syntax + evaluation
  • nix build .#nixosConfigurations.*.config.system.build.toplevel full closure
2. Test Individual Tools
  • nix run .#myNvim test wrapped editor
  • nix run .#myBash test shell config
  • nix run .#myWezterm test terminal
3. VM Testing (QEMU)
  • nixos-rebuild build-vm --flake .
  • Full desktop in a VM window
  • Requires KVM kernel module
4. Deploy When Ready
  • nixos-rebuild switch --flake .
  • Only after steps 1-3 pass
  • Docker/Podman not viable for full NixOS
Key Insight: wrapper-modules are the cross-platform bridge. Traditional Nix configs share user-level settings via home-manager, but application configuration (keybindings, themes, plugins) often lives in NixOS or home-manager modules with platform-specific option names. Wrapper-modules produce standalone derivations — the same myEditor package installs identically on NixOS and darwin without touching any module system.
